# Simple Files Compressor (SFC) 开发日志

---

## 项目概述

**SFC.exe** 是一个基于 **Huffman 编码与 AES-128 加密**实现的文件归档工具，支持对**多个文件与目录进行压缩与解压缩**。项目主要用于研究文件系统归档结构、数据压缩以及基础加密实现。

**核心特性**：
- Huffman 无损压缩，最佳压缩率约 60%
- AES-128 对称加密，SHA-256 密钥派生
- 8MB 分块处理，满载内存峰值控制在 100MB

**关键修复**：
- **`directoryOffset` 计算顺序错位**
  - **问题**：`writeDirectory` 在写入目录内容后才获取文件位置，导致 `directoryOffset` 反映的是目录结束位置而非开始位置
  - **原因**：`EntryProcessor` 写入大量数据后，`tellp()` 返回的位置已经移动到文件末尾
  - **解决**：在 `writeDirectory` 中，**先 flush 并获取当前位置**，再调用 `entryProcessor` 写入目录内容

**修复前后对比**：
```cpp
// 修复前（错误）：目录写入后才获取位置
void HeaderWriter_v0::writeDirectory(...)
{
    EntryProcessor begin(outFile);
    begin.entryProcessor(filePathToScan, fullOutPath, logicalRoot);

    outFile.flush();
    std::streampos endPos = outFile.tellp(); // 此时已是目录结束位置
    // ...
}

// 修复后（正确）：目录写入前获取位置
void HeaderWriter_v0::writeDirectory(...)
{
    StandardsWriter standardWriter;
    Locator locator;

    // 先获取当前位置（目录开始位置）
    outFile.flush();
    std::streampos endPos = outFile.tellp();
    Y_flib::DirectoryOffsetSize directoryOffset = static_cast<Y_flib::DirectoryOffsetSize>(endPos);

    // 再写入目录内容
    EntryProcessor begin(outFile);
    begin.entryProcessor(filePathToScan, fullOutPath, logicalRoot);

    // 回填偏移量
    // ...
}
```

### 2026-04-17：策略模式重构 — 压缩/加密模块抽象化

**目标**：借助接口抽象化调用流程，使得可以动态切换不同压缩或加密算法模块，并通过文件头策略号自动识别。

#### 新增文件

| 文件 | 功能 |
|------|------|
| `NullCompression.h/.cpp` | `ICompression` 空实现，数据直接透传（9字节树桩标记 + 原始数据） |
| `NullEncryption.h` | `IEncryption` 空实现，加解密为恒等拷贝（全 inline，无 .cpp） |
| `StrategyFactory.h/.cpp` | 策略工厂：根据模式创建 `ICompression` + `IEncryption` 模块对 |

#### 策略枚举设计

```cpp
// FileLibrary.h
enum class CompressionMode : uint8_t
{
    HuffmanAES  = 0,  // 默认：Huffman压缩 + AES加密（向后兼容）
    HuffmanOnly = 1,  // 仅Huffman压缩，无加密
    AESOnly     = 2,  // 仅AES加密，无压缩
    PackOnly    = 3   // 仅打包，无压缩无加密
};
```

策略号写入文件头 `Header.strategy` 字段（原已定义但始终为0），解压时自动读取并选择对应模块。

#### 工厂模式

```cpp
// StrategyFactory.h
struct StrategyModules {
    std::unique_ptr<ICompression> compression;
    std::unique_ptr<IEncryption>  encryption;
};

static StrategyModules createModules(CompressionMode mode, const std::string &password);
```

工厂内部使用 `OwnedAesEncryption` 包装类管理 `Aes` 对象生命周期（`AesEncryption` 持有裸指针不拥有，需要外层管理）。

#### NullCompression 树桩设计

管线流程始终为 `[Separated][树块][Separated][数据块]`，NullCompression 的"树块"为 9 字节固定格式：

```
| 0x00 标记 (1B) | 原始数据大小 uint64_t (8B) |
```

解压时读取此 9 字节桩即可判断为无压缩模式，数据块直接透传。每 8MB 块仅增加 9 字节开销（~0.0001%）。

#### 核心管线重构

**BinaryStandardLoader**：所有 `Aes&` 参数改为 `IEncryption&`，`encryptHeaderBlock` 增加模式判断——非加密模式下目录块保持明文（no-op），已有的全零 IV 检测逻辑自动跳过解密。

```cpp
// 新签名
```

**DecompressionLoop**：

```cpp
// 新签名

```

移除了 `processFile` 和 `decompressionLoop` 内部的冗余 `HuffmanCompression` / `AesEncryption` 实例化（原存在两处重复创建）。

**HeaderWriter**：`writeHeader` 和 `headerWriter` 增加 `CompressionMode mode` 参数，将策略号写入文件头。

### 解压中文路径 Bug 修复（v2.1.0）

**问题**：解压时，若输出目录包含中文字符，解压出的文件名乱码。

**原因**：`DecompressionLoop` 构造函数中，用户指定的输出目录直接用 `std::filesystem::path(outputDirectory)` 构造，未经过 `PathTransfer::transPath()` 编码转换。UTF-8 字符串被 `std::filesystem::path` 按本地代码页（GBK）解释，导致路径乱码。

**修复**：

```cpp
// DecompressionLoop.h - 构造函数
// 修复前
parentPath = std::filesystem::path(outputDirectory);
// 修复后
parentPath = transfer.transPath(outputDirectory);
```

**修改文件**：

| 文件 | 修改内容 |
|------|----------|
| `DecompressionLoop.h` | 构造函数中输出目录使用 `transPath()` 转换 |

## 技术难点与解决方案

### 1. Huffman 单字符问题

**问题**：数据块只包含一种字符时，最小堆无法正确合并。

**解决**：检测单字符情况，直接编码为 "0"。

### 2. 目录块边界对齐问题

**问题**：目录数据恰好 16KB 对齐时出现空文件。

**解决**：补充迭代流程，确保边界正确处理。

### 3. 长路径支持（现已放弃）

**问题**：Windows 默认路径长度限制 260 字符。

**解决**：添加 `\\?\` 前缀支持长路径。

### 4. LTO 编译问题

**问题**：启用链接时优化后程序异常。

**解决**：仅使用 O3 优化，禁止 LTO。

### 5. 偏移量动态维护

**问题**：文件流缓冲导致 `tellg/tellp` 更新延迟。

**解决**：软件层动态维护 `tempOffset` 变量。
